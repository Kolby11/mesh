# Requirements Archive: MESH v1.1 Backend Plugin MVP

**Archived:** 2026-05-05
**Original milestone:** v1.1 Backend Plugin MVP
**Outcome:** Shipped

## Outcome Summary

- 28 of 28 v1.1 requirements shipped.
- `BHOST-02` shipped by accepted Phase 03 override: public `mesh.exec_shell` was removed and the MVP contract standardized on structured `mesh.exec(program, args)`.
- Validation metadata cleanup and live-host confirmation remain deferred follow-up work, not missing milestone functionality.

## v1.1 Requirements

### Plugin Package Manifest

- [x] **PINST-01**: The shell has one package.json-like manifest that lists user-installed frontend plugins and backend plugins.
- [x] **PINST-02**: Frontend plugin entries can declare required backend plugin dependencies or backend service categories they need.
- [x] **PINST-03**: Backend plugin entries declare their category/service, such as `audio`, `network`, `power`, `media`, or `shortcuts`.
- [x] **PINST-04**: The package manifest supports multiple backend providers in the same category and records the user's active provider choice.
- [x] **PINST-05**: The shell can parse the package manifest into a normalized installed-plugin graph used by later lifecycle/provider selection work.
- [x] **PINST-06**: The package manifest design remains local-first and does not require remote download, signing, or marketplace behavior in this phase.

### Backend Plugin Lifecycle

- [x] **BPLUG-01**: Backend plugin manifests declare service entrypoints and are validated before runtime launch.
- [x] **BPLUG-02**: Backend plugin discovery creates one runtime instance per enabled backend provider without duplicate stale tasks.
- [x] **BPLUG-03**: Backend Luau `init()` runs once with plugin-scoped context before polling or command dispatch.
- [x] **BPLUG-04**: Backend poll loops run predictably and respect plugin-controlled poll interval changes.
- [x] **BPLUG-05**: Backend runtimes can stop or restart without leaving old poll loops or command receivers active.

### Backend Host API

- [x] **BHOST-01**: Backend Luau scripts can call `mesh.exec(cmd, args)` with structured arguments and receive stdout, stderr, and exit status.
- [x] **BHOST-02**: Backend Luau scripts use strict structured `mesh.exec(program, args)` execution; public `mesh.exec_shell(cmd)` is removed from the MVP surface by the accepted Phase 03 override.
- [x] **BHOST-03**: Backend Luau scripts can call `mesh.config()` and receive plugin settings as a Luau table.
- [x] **BHOST-04**: Backend Luau scripts can call `mesh.log(level, msg)` and produce structured plugin-scoped log entries.
- [x] **BHOST-05**: Backend Luau scripts can call `mesh.service.set_poll_interval(ms)` and affect future poll cadence without shell restart.

### Service Provider Contract

- [x] **BSVC-01**: Backend plugins declare provided service interfaces and provider IDs in manifest/interface metadata.
- [x] **BSVC-02**: `mesh.service.emit(payload)` publishes JSON-compatible state associated with the correct service provider.
- [x] **BSVC-03**: The shell stores the latest emitted backend state for delivery to service consumers.
- [x] **BSVC-04**: Service command requests route to backend Luau command handlers without service-specific Rust branches.
- [x] **BSVC-05**: Service command responses or failures are visible to the caller and diagnostics pipeline.

### Diagnostics and Failure Handling

- [x] **BDIAG-01**: Invalid backend manifests, missing entrypoints, and missing interface contracts produce clear diagnostics.
- [x] **BDIAG-02**: Backend Luau `init()`, poll, emit, and command handler failures degrade plugin health without crashing the shell.
- [x] **BDIAG-03**: Repeated backend failures are deduplicated or rate-limited enough to avoid diagnostic spam.
- [x] **BDIAG-04**: Backend plugin logs and diagnostics include plugin identity and enough context to debug the failing lifecycle stage.

### MVP Proof

- [x] **BREF-01**: A fresh reference backend service plugin exercises config, logging, polling, exported top-level state snapshots, and command handling.
- [x] **BREF-02**: Automated tests prove the reference backend plugin exports state snapshots and handles at least one command through the public backend MVP contract.
- [x] **BREF-03**: Backend MVP behavior is documented in a short reference note sufficient for plugin authors to follow the proven pattern.

## Future Requirements

### Developer Tooling

- **LSP-01**: `.mesh` and Luau plugin LSP completions cover service APIs and backend host APIs.
- **LSP-02**: Hover documentation reflects the backend MVP reference.

### Distribution

- **PKG-01**: Plugin package manager supports installing third-party backend plugins.
- **PKG-02**: Signed or sandboxed plugin packages protect users from untrusted plugin code.

### Frontend and Surfaces

- **FRONTDOC-01**: Full frontend scripting API documentation is validated against a reference frontend component.
- **SURFACE-01**: Additional shell surfaces consume backend services through the stabilized MVP contract.

## Out of Scope

| Feature | Reason |
|---------|--------|
| Remote plugin download and hot install | Local package manifest and installed-plugin graph had to stabilize first. |
| LSP completions and hover | Tooling should document stable APIs, not APIs still being locked. |
| New frontend shell surfaces | This milestone focused on backend plugin fundamentals. |
| Sandboxing/signing untrusted plugins | Security distribution model is a later milestone. |
| Service-specific Rust command behavior | Backend service behavior belongs in Luau plugins. |
