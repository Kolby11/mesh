# Milestone v1.1: Backend Plugin MVP

**Status:** ✅ SHIPPED 2026-05-05
**Phases:** 1-6
**Total Plans:** 21

## Overview

`v1.1` shipped the backend plugin MVP for MESH. The shell now owns one installed-plugin package manifest, derives active backend providers from that graph, runs Luau backend plugins through a stable lifecycle, exposes a locked MVP host API, routes service state and commands through generic contracts, and surfaces backend failures through diagnostics and tests.

## Phases

## Milestone Summary

**Deferred Follow-up:**

- Validation metadata cleanup planned after functional milestone completion and not shipped as part of `v1.1`

**Key Decisions:**

- The shell-owned installed-plugin package manifest is the source of truth for frontend/backend plugin relationships and provider choice.
- Backend service behavior remains in Luau plugins; Rust core stays a generic wiring layer.
- Public backend process execution ships only as strict structured `mesh.exec(program, args)`.
- Service state and command routing use generic contracts instead of service-specific Rust branches.

**Issues Resolved:**

- Replaced scattered backend discovery assumptions with explicit graph-derived provider startup.
- Removed public `mesh.exec_shell` from the backend MVP surface.
- Replaced stale service event storage with latest active-provider state snapshots.
- Added visible backend lifecycle diagnostics and a reference backend proof path.

**Issues Deferred:**

- Nyquist validation files for phases 01-05 remain partial rather than finalized.
- Real host-service startup against live PipeWire or PulseAudio remains a manual confirmation step.
- A stale `latest_service_events` verification note still needs archival cleanup.

**Technical Debt Incurred:**

- Validation metadata cleanup was deferred beyond the functional milestone boundary.
