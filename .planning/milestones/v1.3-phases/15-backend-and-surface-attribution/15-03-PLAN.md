# Plan 15-03 - Publish Delivery and Per-Surface Rollup Stability

## Objective

Record backend state publish/delivery work at the accepted service-event seam and lock the snapshot rollups so per-surface module attribution remains stable beside the new backend view.

## Design Truths

- Only accepted provider updates should contribute publish/delivery samples.
- Surface snapshots must stay deterministic and easy to compare with shell totals.
- Backend and surface rollups belong in the same debug snapshot.

<threat_model>
T-15-03: Publish/delivery timing can be recorded for stale updates or surface rollups can drift while backend data is added, making the resulting inspector data misleading.
Mitigation: reuse the accepted-provider checks in `record_latest_service_state(...)`, record publish/delivery timing around the real fanout path, and add tests for deterministic surface/backend rollups.
</threat_model>
