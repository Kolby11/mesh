# Plan 17-01: Benchmark Contract and Debug Snapshot Shape

<objective>
Add the shell-owned benchmark scenario/result contract to the existing debug snapshot and expose safe default benchmark rows through `mesh.debug`, without adding launch behavior or UI changes yet.
</objective>

<requirements>
- `BENCH-01`: Hover scenario exists with input/restyle/render timing fields.
- `BENCH-02`: Surface open/close scenario exists with render/redraw fields.
- `BENCH-03`: Pointer-driven update scenario exists with input-to-visible-response fields.
- `BENCH-04`: Keyboard traversal scenario exists with focus/input/render fields.
- `BENCH-05`: Backend-driven update scenario exists with backend/frontend correlation fields.
</requirements>

<decision_coverage>
- `D-02`: stable scenario ids, labels, targets, and statuses are defined in typed debug data.
- `D-04`: debug profiling remains independently toggled; this plan does not add launch behavior.
- `D-09`: result summaries use existing live rolling profiling buckets.
- `D-10`: benchmark rows carry scenario id, target, status, and timing summaries.
- `D-11`: no persistence, history, trace export, or replay is introduced.
</decision_coverage>

<threat_model>
| Threat | Severity | Mitigation |
|--------|----------|------------|
| Benchmark state accidentally enables profiling or emits profiling data while disabled | high | Keep benchmark data derived from debug/profiling state; tests must assert profiling remains disabled and `profiling` remains null when disabled. |
| Scenario ids drift from fixed benchmark suite | medium | Define scenario ids in typed Rust enum/constant helpers and assert exact JSON ids in shell tests. |
| Benchmark contract becomes trace/history storage | medium | Only add latest live snapshot/result fields; do not add file paths, history arrays, timestamps beyond existing sample order, or persistence APIs. |
</threat_model>
