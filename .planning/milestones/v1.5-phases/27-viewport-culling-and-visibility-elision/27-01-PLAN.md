# Plan 27-01 - Viewport-aware retained paint omission with explicit hidden semantics

## Objective

Reduce retained paint traversal on the software renderer by omitting fully invisible subtrees only where explicit hidden state or explicit viewport authority exists, while keeping generic opacity semantics and the existing debug/profiling contract intact.

## Design Truths

- D-01: Use Qt-like conservative visibility semantics rather than a global smart culling pass.
- D-02: Do not treat plain `opacity: 0` as a general hidden-state shortcut in this phase.
- D-03: Explicit clip and scroll viewport boundaries are the authority for subtree omission.
- D-06: Localized rough pre-clipping is allowed for viewport-aware content, but partial intersections still paint.
- D-09: Proof stays aggregate-only in the existing invalidation/debug payload.

<threat_model>
T-27-01: Over-aggressive pruning could silently drop visible content, conflate generic opacity with explicit hidden semantics, or create a second debug/proof path that drifts from the retained paint contract.
Mitigation: separate explicit hidden semantics from opacity, constrain omission to explicit viewport/clip authority plus trivial root-surface exclusion, keep partial intersections paintable, and expose only aggregate counters through the existing profiling/debug payload with focused render and shell tests.
</threat_model>

## Risks

- The current style system collapses `visibility:hidden` into `opacity = 0.0`; changing that seam can affect both render and interaction assumptions if not kept narrow.
- Over-pruning partially visible descendants would create correctness regressions that Phase 27 is explicitly supposed to avoid.
- Extending debug payloads too far would expand scope into diagnostics work rather than a bounded optimization phase.
