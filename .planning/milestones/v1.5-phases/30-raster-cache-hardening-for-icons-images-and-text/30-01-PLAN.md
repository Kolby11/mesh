# Plan 30-01 - Renderer-owned raster cache hardening and proof

## Objective

<objective>
Remove repeat rasterization, resize, parse, and avoidable shaping cost from steady-state CPU painting by adding renderer-owned raster variant caches for SVG/bitmap/missing icons, preserving and proving text/glyph reuse, and carrying conservative opaque/translucent metadata through existing profiling proof.
</objective>

## Design Truths

- Keep raster caches in `mesh-core-render`; shell remains the profiling consumer, not the cache owner.
- Cache rendered variants, not just decoded source assets.
- Use conservative keys and freshness checks so correctness wins over hit rate.
- Treat unknown alpha as translucent and only use proven opacity for blend/background elision.
- Extend existing profiling proof instead of adding another diagnostics surface.

<threat_model>
T-30-01: Raster caches can serve stale or visually wrong resources if keys omit dimensions, tint, multicolor mode, icon axes, or file freshness. Opaque metadata can cause under-blending if inferred too aggressively. Text/glyph changes can regress shaping correctness if cache keys omit layout-affecting inputs or if proof only covers isolated unit calls. Mitigation: key caches by all visual inputs, invalidate/bypass on uncertain file freshness, inspect alpha for opacity metadata, use opacity only for conservative painter/display-list decisions, keep fallbacks unchanged, and add focused cache-hit/miss tests plus shell interaction profiling proof.
</threat_model>

<task id="30-01-01" wave="1">
<title>Add renderer-owned raster variant cache for SVG, bitmap, and missing icons</title>
<read_first>
- `crates/core/frontend/render/src/surface/icon.rs`
- `crates/core/frontend/render/src/surface/buffer.rs`
- `crates/core/frontend/render/src/surface/profiling.rs`
- `.planning/phases/30-raster-cache-hardening-for-icons-images-and-text/30-CONTEXT.md`
- `.planning/phases/30-raster-cache-hardening-for-icons-images-and-text/30-RESEARCH.md`
</read_first>
<action>
1. In `surface/icon.rs`, add a bounded raster variant cache for file-backed SVG/bitmap icons/images and the built-in missing-icon fallback.
2. Key file-backed variants by canonical path when `std::fs::canonicalize` succeeds; otherwise use a normalized absolute/path identity derived from the input path without collapsing distinct unresolved paths.
3. Store and compare file length plus modified timestamp when both are available. If metadata cannot be read or cannot be trusted, bypass the file-backed cache for that paint rather than serving a potentially stale entry.
4. Key the built-in missing icon separately by destination width/height and tint.
5. Store rendered pixels and conservative opacity metadata with each cached variant.
6. On a cache hit, blit cached pixels into the destination buffer without recording new `icon_image_raster_micros`.
7. On a cache miss, preserve the existing decode/resize/SVG/missing-icon behavior and record raster time.
8. Keep missing/unsupported asset fallbacks unchanged.
</action>
<acceptance_criteria>
- Repeated SVG paints with identical visual inputs reuse a cached raster variant and do not increase `icon_image_raster_micros` on the second paint.
- Repeated bitmap paints with identical visual inputs reuse the resized/tinted variant and do not resize again on the second paint.
- Repeated missing-icon fallback paints reuse a cached rendered variant.
- Tint and multicolor mode produce distinct cache entries.
- File metadata changes invalidate or bypass stale file-backed variants.
- File-backed cache is bypassed when length/modified metadata cannot be read.
- Cached variants retain a conservative fully-opaque vs translucent/unknown flag.
</acceptance_criteria>
