# Plan 31-01 - Conservative CPU smoothness tuning and shipped-surface proof

## Objective

<objective>
Tune the retained CPU rendering heuristics conservatively, prove the five canonical shipped-surface scenarios are visibly smoother, and record the remaining GPU or parallel-rendering boundaries without introducing a new renderer architecture, benchmark system, or shell UI surface.
</objective>

## Design Truths

- Use existing Phase 26 and Phase 30 proof as the comparison baseline.
- Keep repaint-policy selection in shell orchestration and retained command selection in `mesh-core-render`.
- Prefer leaving thresholds conservative over accepting a counter-only improvement that cannot prove visual correctness.
- Publish proof through existing profiling/debug payloads and planning artifacts.
- Treat manual UAT as required acceptance evidence, not an optional note.

<threat_model>
T-31-01: Repaint-policy tuning can underpaint, skip required backgrounds, lose scrollbar or tooltip overlays, or reorder commands if minimal damage is over-selected. Mitigation: keep `RetainedDisplayList::select_paint_commands` ordering and fallback tests green, add threshold tests around `select_damage_policy`, and require UAT correctness notes for every canonical scenario.

T-31-02: Benchmark-only tuning can optimize counters while users still perceive lag. Mitigation: require `31-01-BENCHMARK.md` and `31-UAT.md`; a tuning change is accepted only when the benchmark table and UAT row both support it.

T-31-03: Cache capacity tuning can hide stale resources, weaken SVG external-resource bypasses, or increase memory without measurable smoothness benefit. Mitigation: only change capacity constants through explicit threshold gates, preserve freshness/bypass/key tests, and document unchanged capacities when evidence does not justify modification.

T-31-04: Scope creep into GPU, parallel paint/layout, a new benchmark harness, or shell UI redesign would invalidate the milestone boundary. Mitigation: keep touched files inside the listed CPU retained-rendering and proof artifacts, and make SMTH-03 an explicit verification item.
</threat_model>
