# Requirements Archive: MESH v1.19

**Defined:** 2026-06-09
**Completed:** 2026-06-09
**Core Value:** MESH should let plugin authors build distinctive shell UI and service integrations while the shell stays observable, deterministic, and responsive on real interaction paths.

## v1.19 Requirements (All Satisfied)

### Scheduler

- [x] **SCHED-01**: Shell blocks on Wayland connection fd via `poll()` with deadline from `next_runtime_sleep()` instead of `std::thread::sleep` when running the Wayland surface backend
- [x] **SCHED-02**: Backend/command message senders signal an eventfd to wake the blocking Wayland poll, preventing stale-frame latency on IPC events
- [x] **SCHED-03**: Shell loop preserves drain-first ordering: drain shell messages → tick → render → present → block
- [x] **SCHED-04**: Dev-window backend (minifb) retains its existing sleep path — deadline-driven blocking applies to the Wayland surface backend only

### Opaque Region

- [x] **OPAQUE-01**: Present path walks the retained display list to find fully-opaque root background rects, computes their union as a `wl_region`, and sends `wl_surface::set_opaque_region`

### Diagnostics

- [x] **DIAG-01**: Profiling infrastructure records `ProfilingStage::SchedulerIdle` with block duration and wake reason, visible in debug inspector

## v2 Requirements (Deferred)

- **DIAG-02**: Structured `mesh.debug.scheduler` payload
- **OPAQUE-02**: Conservative opaque-region gating for border-radius, shadows, effects

## Out of Scope

| Feature | Reason |
|---------|--------|
| tokio-based async event loop | Shell loop is synchronous by design |
| calloop integration | Existing `rustix::event::poll` is simpler |
| Compositor-specific opaque region hints | Standard protocol works on all compositors |
| Multi-fd epoll integration | Only needed with timerfd or additional sockets |
| wp_presentation feedback | Deferred until frame-scheduler stability is proven |
