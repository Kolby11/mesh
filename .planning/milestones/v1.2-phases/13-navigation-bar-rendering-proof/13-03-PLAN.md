# Plan 13-03 - Motion Proof and Interaction Clarity on the Shipped Bar

## Objective

Apply the Phase 12 animation contract and Phase 10/11 interaction boundaries to the richer nav bar so the surface visibly proves motion, selection, and focus behavior without becoming noisy or ambiguous.

## Design Truths

- Token-driven transitions should polish the whole surface.
- Only one clear custom keyframe moment is approved.
- The strongest visual interactive emphasis must remain focus and active state, not motion.

<threat_model>
T-13-03: A custom-keyframe proof can easily obscure focus clarity or create a misleading impression that controls and passive status text share the same interaction model.
Mitigation: keep the keyframe proof on one bounded status/accent element, leave button focus styles dominant, and constrain `selectable="true"` to passive text only.
</threat_model>

<steps>
1. Replace any remaining raw motion timing values in nav-bar styles with `token(animation.*)` values where practical for transitions and animation declarations.
2. Add exactly one custom `@keyframes` proof in `modules/frontend/navigation-bar/src/main.mesh` or a single local component, using percentage-only stops and supported visual properties.
3. Keep the custom keyframe attached to one bounded status accent, chip, or utility state moment; do not animate the full bar container continuously.
4. Preserve or refine existing hover/focus/active transitions on the existing controls so they read as one surface family after the migration.
</steps>

</acceptance_criteria>

<task id="13-03-02" wave="3">
<title>Keep selection, focus, and control semantics visually distinct</title>
<read_first>
- `modules/frontend/navigation-bar/src/main.mesh`
- `modules/frontend/navigation-bar/src/components/settings-button.mesh`
- `modules/frontend/navigation-bar/src/components/theme-button.mesh`

- `.planning/phases/11-keyboard-navigation-and-shortcuts/11-CONTEXT.md`
</read_first>
<steps>
1. Ensure passive selectable status text remains outside the clickable control wrappers and does not inherit button hover/focus/active styling.
2. Preserve strong focus-visible styling on actual controls so keyboard focus remains more visually authoritative than passive status styling.
3. If visible control-adjacent values are introduced, keep them secondary to the button-first model and non-selectable unless they are part of the approved passive status cluster.
4. Avoid visual treatments that make the status cluster look like a row of pseudo-links or pseudo-buttons.
</steps>
<acceptance_criteria>
- `modules/frontend/navigation-bar/src/main.mesh` keeps `selectable="true"` only on passive text nodes.
- Control component files still include their focus/active selectors after the migration.
- No nav-bar component file adds `selectable="true"` to button wrappers.
</acceptance_criteria>
