# Plan 24-01: Typed Selector Inputs and Rule Index

## Goal

Add typed selector inputs and a conservative rule index to reduce repeated string scans in style resolution while preserving existing string-map compatibility and full-scan fallback behavior.

## Tasks

### 24-01-01: Typed Selector Inputs

- Add `StyleNodeAttrs` or equivalent typed helper in `style/resolve.rs`.
- Extract class tokens, id, key, module id, tag, and state once per node/restyle call.
- Route selector matching through typed attributes while preserving existing public resolver APIs.

### 24-01-02: Selector Rule Index

- Add `StyleRuleIndex` for tag, class, id/key, and pseudo-state selector dependencies.
- Use the index for candidate rule lookup when selectors are known/simple.
- Fall back to full scans for universal, unsupported, or compound selectors with unknown dependencies.
- Add tests proving indexed and full-scan results match.

### 24-01-03: Retained Restyle Integration

- Use typed attributes and indexed rule candidates in `restyle_subtree` and `restyle_subtree_for_keys`.
- Keep current behavior for diagnostics and theme defaults.
- Record summary and verification artifacts.

## Verification

- `cargo fmt --check`
- `cargo test -p mesh-core-elements style_rule_index`
- `cargo test -p mesh-core-elements restyle_subtree`
- `cargo test -p mesh-core-elements`

## Risks

- Selector indexing must not skip compound/state rules. Unknown dependencies must fall back to full rule scans.
- Existing string attributes remain the compatibility source, so typed helpers must not mutate node data.
