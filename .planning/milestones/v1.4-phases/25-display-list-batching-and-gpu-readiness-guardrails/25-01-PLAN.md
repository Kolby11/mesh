# Plan 25-01: Display-List Batching Metrics and GPU Guardrails

## Goal

Extend the retained display-list layer with conservative batching summaries, explicit barrier reasons, and GPU-readiness proof while keeping the software painter unchanged.

## Tasks

### 25-01-01: Batching Summary

- Add typed batching barrier reasons to `mesh-core-render`.
- Compute adjacent batch summaries from retained display-list entries.
- Batch only same-signature compatible primitives that are not text/icon/opacity/clip/translucency barriers.
- Count batches, merged primitives, barriers, and barrier reason occurrences.

### 25-01-02: Debug Metrics

- Extend `RetainedPaintSnapshot` with batching metrics and readable barrier reason counters.
- Serialize batching metrics in debug JSON under `invalidation.paint`.
- Keep metrics observational; do not alter paint ordering or issue GPU commands.

### 25-01-03: Proof and Handoff

- Add focused unit tests for batching and barriers.
- Add GPU readiness proof documentation for future GPU and parallel paint/layout milestones.
- Run render and shell profiling verification.

## Verification

- `cargo fmt --check`
- `cargo test -p mesh-core-render display_list_batch`
- `cargo test -p mesh-core-render`
- `nix develop -c cargo test -p mesh-core-shell profiling_snapshot_exposes_typed_surface_invalidation_counts`

## Risks

- Batching must not imply visual merging where barriers exist. Metrics only describe safe opportunities.
- GPU readiness must stay at the data/guardrail level and not introduce backend abstractions prematurely.
