# Phase 16 - UI Design Contract

> Visual and interaction contract for Phase 16 debug-only profiling mode and live inspector. Generated by `gsd-ui-phase`, verified inline against the six UI-SPEC dimensions.

---

## Scope

Phase 16 replaces the current native right-side debug panel with a shell-shipped `.mesh` inspector surface that lives inside the existing debug path.

The approved UI proof is not a new diagnostics product surface, not an end-user settings screen, and not a benchmark launcher. The approved surface must prove:

- the debug inspector is rendered with normal `.mesh` UI primitives
- the familiar right-side debug panel interaction model survives the implementation change
- profiling enable remains explicit and separate from inspector visibility
- overview, surfaces, backend services, and benchmark/interaction views all exist in one coherent inspector
- sparse or empty profiling data produces stable, readable zero states instead of broken UI

This UI contract does not approve:

- a separate profiling-only entrypoint
- a fullscreen dashboard treatment
- trace persistence or replay UX
- Phase 17 benchmark execution flows
- broad restyling of unrelated shell chrome

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | none - native `.mesh` elements only |
| Icon library | XDG icon names rendered through MESH `<icon>` where helpful for tabs, states, and metric categories |
| Font | `token(typography.family)` / Inter |
| Theme source | active shell theme tokens plus existing dark/light shell themes |

### Non-Negotiable Design Rules

- Use theme tokens only: `token(color.*)`, `token(spacing.*)`, `token(radius.*)`, `token(typography.*)`, and `token(animation.*)`.
- Do not introduce raw hex colors in `.mesh` implementation files.
- Keep the inspector recognizably shell tooling: dense, operational, and legible, not dashboard-marketing or IDE-chrome parody.
- The inspector must remain a right-side panel, not a floating center modal or a fullscreen page.
- Profiling controls must read as explicit debug controls; opening the inspector must not visually imply that profiling is already running.
- Empty and no-sample states are first-class design states, not fallback afterthoughts.
- Benchmark content in Phase 16 must explain categories and intent only; it must not masquerade as runnable benchmark automation.

## Surface Contracts

### Primary Inspector Surface

The Phase 16 proof surface is one debug-only right-side inspector panel rendered through `.mesh`.

| Element | Contract |
|---------|----------|
| Host role | Replaces the native debug panel while preserving the same edge-mounted debug-panel mental model. |
| Placement | Right edge of the shell, vertically spanning the debug panel area. |
| Tone | Tooling-first, compact, information-dense, and calm under live updates. |
| Ownership | Shell-shipped and debug-only, but authored as a normal `.mesh` module/surface. |
| Primary sections | Header/status strip, profiling control zone, view navigation, and scrollable view body. |
| Width behavior | Narrow enough to coexist with active shell surfaces, wide enough to keep metric labels and empty-state copy readable. |

### Header And Status Zone

| Element | Contract |
|---------|----------|
| Inspector title | Short operational title such as “Profiler” or “Debug Inspector,” not a marketing heading. |
| Session state | Clearly distinguishes “profiling off,” “profiling live,” and “profiling on but collecting sparse data.” |
| Session ID / live badge | Allowed if compact and secondary to the state message. |
| Control tone | Toggle and mode controls must read as shell tooling, not as a consumer-facing preferences UI. |

### View Navigation

Phase 16 must expose exactly four top-level inspector views.

| View | Contract |
|------|----------|
| Overview | High-level shell summary, profiling state, headline timing buckets, and whether current samples are meaningful yet. |
| Surfaces | Per-surface timing breakdowns with stable surface identity and readable empty states for surfaces with no recent work. |
| Backend Services | Provider/service lifecycle plus backend stage timing summaries, clearly separated from frontend surface cost. |
| Benchmark / Interaction | Scaffolded category cards for hover, open/close, pointer-driven update, keyboard traversal, and backend-driven update. |

Navigation constraints:

- Use compact segmented tabs, pills, or a stacked nav list inside the panel.
- Navigation labels must stay visible; icon-only navigation is not approved as the primary model.
- Switching views must not implicitly enable profiling.

### Overview View

| Element | Contract |
|---------|----------|
| Top summary | Shows whether profiling is off, warming up, or actively collecting recent samples. |
| Metric grouping | Shell-wide timing buckets should be grouped into readable operational sections rather than one long raw list. |
| Headline numbers | Emphasize comparability and trend-at-a-glance, not giant dashboard numerics. |
| Explanation | If no samples exist, explain why and how to obtain them using the existing debug path. |

### Surfaces View

| Element | Contract |
|---------|----------|
| Row identity | Each row must preserve `surface_id` and, where useful, module identity. |
| Row content | Surface label, headline timing, redraw activity, and optionally the most relevant recent stage breakdown. |
| Empty rows | Surfaces with no recent samples render as stable “no recent activity” rows or cards, not missing containers. |
| Comparison tone | The view should help compare individual surfaces against shell-wide totals without pretending to be a full trace explorer. |

### Backend Services View

| Element | Contract |
|---------|----------|
| Row identity | Each row must preserve interface and provider identity. |
| Separation of concerns | Runtime health and profiling cost appear together only if visually separated; health must not be mistaken for performance cost. |
| Stage display | Poll/update, command handling, and state publish/delivery should be legible as distinct backend stages. |
| Empty state | Missing or idle providers render as factual zero/idle states without looking broken or alarming. |

### Benchmark / Interaction View

| Element | Contract |
|---------|----------|
| Goal | Establish the information architecture for Phase 17 benchmark flows. |
| Category cards | Must include hover, surface open/close, pointer-driven update, keyboard traversal, and backend-driven update. |
| Copy | Each category explains what it will measure once benchmark flows exist. |
| Status | Each category should read as scaffolded / pending, not interactive and incomplete. |

## Responsive Contract

Although the inspector is a fixed debug panel concept, it still needs intentional compact behavior inside a constrained width.

| Width Pressure Rule | Contract |
|---------------------|----------|
| First compression step | Secondary helper copy compresses or wraps before primary labels disappear. |
| Metric preservation | Primary labels, state badges, and essential timing values stay visible longer than tertiary explanatory text. |
| Navigation preservation | All four view labels remain understandable in compact width; do not collapse immediately to icon-only tabs. |
| Dense data strategy | Prefer stacked rows, truncation of long IDs, and secondary-copy reduction before shrinking control affordances. |
| Scroll behavior | Vertical scrolling in the body is allowed; horizontal overflow for core content is not approved. |

Responsive anti-goals:

- Do not hide profiling state just to save width.
- Do not allow backend/provider identifiers to overflow outside the panel chrome.
- Do not use a multi-column dashboard layout that collapses poorly in the narrow inspector width.

## Spacing Scale

Declared values must use existing tokens and remain multiples of 4.

| Token | Value | Usage |
|-------|-------|-------|
| `spacing.xs` | 4px | Tight metric gaps, inline badge padding, micro-separators |
| `spacing.sm` | 8px | Compact row spacing, button gaps, stacked helper text |
| `spacing.md` | 16px | Default panel padding and section spacing |
| `spacing.lg` | 24px | Major section separation inside the inspector |
| `spacing.xl` | 32px | Optional larger spacing between header/control zone and body on wide panels |
| `spacing.2xl` | 40px | Not required by default |
| `spacing.3xl` | 48px | Not required |
| `spacing.4xl` | 64px | Not required |

Exceptions: none

### Stable Dimensions

| Element | Contract |
|---------|----------|
| Panel width | Target approximately the current native debug panel width class; avoid significant expansion beyond a compact tooling sidebar. |
| Header height | Compact toolbar scale; it should not read like an app page header. |
| Row density | Dense but readable; prefer consistent row heights for metrics and lists rather than highly irregular cards. |
| Card rhythm | Use `spacing.sm` or `spacing.md` between stacked data blocks; avoid oversized breathing room that wastes narrow panel width. |
| Corner language | Use theme radii consistently, with slightly stronger rounding on cards/chips than on structural panel chrome. |

## Typography

Use the existing theme typography. The inspector should feel operational and information-dense rather than editorial.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body metric row | `token(typography.size.sm)` / 12px | 400-500 | `token(typography.line-height.normal)` / 1.4 |
| Secondary helper text | `token(typography.size.xs)` / 10-12px | 400-500 | 1.4 |
| Section label | `token(typography.size.sm)` or `token(typography.size.md)` / 12-14px | 600 | 1.4 |
| Inspector title / key emphasis | `token(typography.size.md)` / 14px | 600-700 | 1.4 |

Typography constraints:

- Keep letter spacing at `0`.
- Do not introduce oversized dashboard numerics or banner-like display text.
- Long identifiers may truncate visually, but the visible typography must stay calm and legible.
- Empty-state copy should stay compact and factual, not verbose instructional prose.

## Color

Use semantic theme tokens, not raw hex values in implementation.

| Role | Token / Value | Usage |
|------|---------------|-------|
| Dominant (60%) | `color.surface`, `color.surface-container-low` | Main inspector panel background and structural chrome |
| Secondary (30%) | `color.surface-container`, `color.surface-container-high`, `color.surface-container-highest` | Metric cards, row groups, view navigation containers, empty-state tiles |
| Accent (10%) | `color.primary`, `color.secondary-container`, `color.tertiary-container` | Active view state, live profiling badge, focus-visible states, selected/high-priority metric emphasis |
| Destructive | `color.error`, `color.error-container` | Error or failed backend/runtime states only |

Accent reserved for:

- active inspector view indicator
- live profiling state badge or explicit “profiling on” emphasis
- keyboard focus-visible states
- the most important current metric or comparison emphasis in a view

Color constraints:

- The full panel background should stay rooted in surface tokens, not become an accent slab.
- Idle metric rows should not compete chromatically with active/focused controls.
- Empty states should look intentionally inactive, not disabled by accident.
- Backend failure/error styling must be clearly distinct from “no recent samples” neutral states.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Not applicable; the inspector is operational tooling, not a CTA surface |
| Empty state heading | Short factual labels such as “Profiling is off”, “No recent surface activity”, or “Benchmarks land in Phase 17” |
| Empty state body | One compact sentence that explains the current state and, when needed, the next debug-path action |
| Error state | Factual problem wording with the affected provider/surface and a concise recovery clue |
| Destructive confirmation | Not applicable for this phase |
| General tone | Debug-tool language: concise, factual, and system-oriented rather than tutorial-heavy |

Copy constraints:

- Avoid promotional language, hype, or faux-product headings.
- Do not imply hidden automation such as “start benchmark now” in Phase 16.
- Avoid blaming language for sparse data; say what is missing and why.
- Prefer concrete state copy over vague adjectives like “healthy,” “fast,” or “good” without context.

## Interaction Contract

| Interaction | Required Behavior |
|-------------|-------------------|
| Open inspector | Uses the existing debug overlay/debug command path and mounts the `.mesh` panel without enabling profiling automatically. |
| Enable profiling | Uses an explicit profiling control or mode inside the existing debug flow; it does not auto-switch views. |
| Switch views | Changes only the current inspector body content and preserves profiling collection state. |
| Focus navigation | Maintains existing shell keyboard/focus-visible expectations for compact debug controls. |
| Sparse data arrival | Live values may populate incrementally without destabilizing layout or reordering the user unexpectedly. |
| Empty-state browsing | Users can still navigate between all four views even when no profiling samples exist. |

### Animation Contract

| Motion Area | Contract |
|-------------|----------|
| General polish | Use token-driven color, border, opacity, and spacing transitions only where they clarify state changes. |
| Live state emphasis | A restrained badge pulse or subtle state change is allowed for “profiling live,” but only on one bounded element. |
| View changes | Prefer immediate or lightly transitioned content changes; do not use theatrical panel slides. |
| Metric updates | Avoid noisy constant motion on every sample refresh. |
