# Plan 16-03 - Overview, Surfaces, and Backend Services Inspector Views

## Objective

Implement the core inspector views and their sparse-data UX so the built-in debug panel becomes a usable `.mesh` consumer of shell profiling state instead of a native placeholder.

## Design Truths

- All view switching is local inspector UI state and must not mutate profiling collection.
- Empty-state copy is required in every view that can receive no recent samples.
- Backend lifecycle status and timing stages must read as related but distinct information.

<threat_model>
T-16-03: The inspector can technically render profiling payloads while remaining unusable because sparse states disappear, backend/runtime concepts blur together, or view switching accidentally becomes coupled to profiling activation.
Mitigation: implement explicit view components with stable zero-state copy, preserve local view state in `.mesh`, and add real-surface tests for off/idle/live data scenarios.
</threat_model>
