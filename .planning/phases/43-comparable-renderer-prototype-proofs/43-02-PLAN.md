# Plan 43-02 - Blitz Reference Prototype Evidence

<objective>
Produce PROTO-01 evidence by attempting a Blitz-based reference prototype for the shared navigation bar and audio popover fixture. If Blitz cannot render the required slice within throwaway scope, record a concrete reproducible blocker that still supports the PROTO-03 comparison.
</objective>

<must_haves>
- D-01: Compare structural and behavioral parity, not pixel-perfect production fidelity.
- D-02: Structured output is acceptable if pixel rendering exceeds scope.
- D-03: Cover hover, click, slider movement/release, and open-close behavior.
- D-04: Use the shared fixture from Plan 43-01.
- D-06: Use HTML/CSS-equivalent input for Blitz rather than `.mesh` translation.
- D-07: Record exact blocker fields if Blitz cannot render.
- D-08: Keep shell/window integration throwaway-only.
- D-10: Use common comparison headings.
</must_haves>

<threat_model>
- T-43-04 Blitz API churn: High. Mitigate by documenting the exact dependency line, crate/API boundary, command, and compile/runtime error when the harness cannot be completed.
- T-43-05 Non-comparable Blitz input: Medium. Mitigate by generating Blitz HTML/CSS from the shared fixture scenario IDs and retained node labels, not from a separate hand-authored demo.
- T-43-06 Over-investing in shell integration: Medium. Mitigate by accepting structured evidence or blocker evidence without production Wayland/layer-shell integration.
</threat_model>
