# Plan 45-01 - Renderer Migration Roadmap

<objective>
Create `docs/renderer-migration.md` as the maintainer-facing roadmap for broad renderer migration, with phased reversible steps and explicit build, CI, feature flag, Linux/Nix, binary-size, and rollback gates before broad adoption.
</objective>

<must_haves>
- D-01: Use phased, reversible adapter expansion.
- D-02: Expand Phase 44 proof boundaries before replacement.
- D-03: Do not approve a whole-renderer rewrite as the first step.
- D-12: Require feature-gated or reversible rollout.
- D-13: Document Linux/Wayland/Nix, workspace dependency, native library, binary-size, build-time, CI, and workspace command implications.
- D-14: Require workspace, proof, shipped-surface, selection, invalidation/damage/profiling, and AccessKit evidence gates.
- D-15: Preserve observability parity before any path becomes authoritative.
- D-18: Cite Phase 42/43 Blitz evidence rather than reopening broad dependency research.
</must_haves>

<threat_model>
- T-45-01-01 Broad rewrite ambiguity: High. Mitigate by including the exact sentence `MIGR-01: broad renderer migration is phased and reversible; whole-renderer rewrite is not the first migration step.`
- T-45-01-02 Hidden dependency cost: High. Mitigate by requiring every migration step to fill fields named `Linux/Nix impact`, `Root workspace dependencies`, `Native libraries`, `Binary/build risk`, `CI gates`, and `Rollback path`.
- T-45-01-03 Lost observability: High. Mitigate by listing `NodeId`, `typed invalidation`, `damage`, `profiling`, `diagnostics`, and `AccessKit-compatible update evidence` as blocking promotion gates.
</threat_model>
