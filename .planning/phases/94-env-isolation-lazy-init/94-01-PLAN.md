### 1d: Add `ensure_initialized()` method
Add this method to `impl ScriptContext` (insert before `load_script`):
```rust
    /// Check out a pooled VM, create a per-component _ENV table, install host API,
    /// sandbox the thread, and populate builtin_globals. Idempotent — does nothing
    /// if already initialized.
    fn ensure_initialized(&mut self) -> Result<(), ScriptError> {
        if self.vm.is_some() {
            return Ok(());
        }
        let vm = pool::checkout();
        let lua = vm.lua();

        // ISO-01: create per-component _ENV with __index = globals() fallthrough
        let env = lua.create_table().map_err(lua_err)?;
        let meta = lua.create_table().map_err(lua_err)?;
        meta.set("__index", lua.globals()).map_err(lua_err)?;
        env.set_metatable(Some(meta)).map_err(lua_err)?;

        self.vm = Some(vm);
        self.env_table = Some(env.clone());

        // Install host API into the per-component env table (ISO-02 completion)
        self.install_host_api(&env)?;

        // Snapshot all keys on env_table post-host-api-install (stdlib keys are
        // on globals, not env_table — only host API and later script globals show up here)
        self.builtin_globals = env
            .pairs::<String, LuaValue>()
            .filter_map(|result| result.ok().map(|(key, _)| key))
            .collect();

        Ok(())
    }
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| env_table ↔ lua.globals() | env_table.__index falls through to read-only pool VM globals — user script code can READ stdlib but can only WRITE to env_table |
| closure ↔ env_table | Cloned Table handle in closures — writes are scoped to this component's private env_table |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-94-01 | Tampering | ScriptContext::env_table | mitigate | env_table has `__index = lua.globals()` metatable only — no `__newindex` override. Writes to env_table go to env_table; globals remain read-only via `Lua::sandbox(true)` on pool VMs. |
| T-94-02 | Information Disclosure | PooledVm checkout | mitigate | Each checkout creates a fresh env_table — `pairs()` only sees keys installed since checkout. Previous component's state is inaccessible because env_table is created new per checkout. |
| T-94-03 | Elevation of Privilege | Closure capture of env_table | accept | Closures capture env_table.clone() (Table handle, cheap). They can write to this component's env_table. This is the intended behavior — request_redraw and locale.current need write access to the component's namespace. No cross-component access possible because each component has a different env_table reference. |
| T-94-04 | Spoofing | Script with set_environment | mitigate | mlua's `Chunk::set_environment()` sets the first upvalue of the compiled chunk, which is Luau's `_ENV` for that chunk. The script CANNOT escape to write lua.globals() because `Lua::sandbox(true)` makes globals read-only. Attempted writes to globals through `_ENV` fallthrough would fail silently or error. |
</threat_model>
