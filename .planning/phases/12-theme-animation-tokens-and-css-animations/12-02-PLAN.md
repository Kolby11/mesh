# Plan 12-02 - Strict Percentage Keyframe Parsing

## Objective

Replace the Phase 8 `@keyframes` rejection boundary with a strict Phase 12 parser that accepts percentage-only keyframes over the transition-safe property set and rejects unsupported, alias-based, or non-runnable rules.

## Design Truths

- Keyframe stops are percentages only, including intermediate percentages.
- `from` and `to` aliases are out of scope for the first release.
- Unsupported properties reject the entire keyframes block.
- Keyframe stop values do not need theme token references in Phase 12.

<threat_model>
T-12-02: Permissive keyframe parsing could accept browser CSS that the renderer cannot honor, making animations silently incomplete.
Mitigation: implement strict percentage-only parsing, validate every stop property against the transition-safe set, and reject unsupported or empty keyframe rules with parser diagnostics.
</threat_model>

<title>Add keyframe data structures to the component/style model</title>
<read_first>
- `.planning/phases/12-theme-animation-tokens-and-css-animations/12-CONTEXT.md`
- `.planning/phases/12-theme-animation-tokens-and-css-animations/12-RESEARCH.md`

- `crates/core/ui/elements/src/style/types.rs`
- `crates/core/ui/elements/src/style/parse.rs`
</read_first>
<steps>
1. Add public keyframe data structures reachable from parsed component files: `KeyframeRule { name, stops }`, `KeyframeStop { offset, declarations }`, and a collection on the parsed style block or component file.
2. Store offsets as normalized `f32` values in the inclusive range `0.0..=1.0`.
3. Store keyframe stop declarations using the existing `Declaration` representation so later resolution can reuse property parsing logic.
4. Add helper logic that identifies the transition-safe animatable property names exactly from the context: background/background-color, border-color, border-radius and corner longhands, border-width and edge longhands, color, opacity, width, height, min/max dimensions, padding and padding longhands, margin and margin longhands, transform, font-size, letter-spacing, line-height, gap, row-gap, column-gap, gap-x, inset, top, right, bottom, and left.
5. Add unit tests for the transition-safe property helper so unsupported examples such as `filter`, `box-shadow`, `grid-template-columns`, and `display` are false.
</steps>
<acceptance_criteria>
- A parsed component style model can carry at least one named keyframe rule.
- `crates/core/ui/elements/src/style/types.rs` or an adjacent style module exposes a transition-safe keyframe property helper.
- Tests prove `opacity`, `transform`, `border-radius`, `padding`, `font-size`, and `inset` are accepted keyframe properties.
- Tests prove `filter`, `box-shadow`, `grid-template-columns`, and `display` are rejected keyframe properties.
</acceptance_criteria>

<task id="12-02-02" wave="1">
<title>Lower strict percentage-only `@keyframes` from Lightning CSS</title>
<read_first>
- `crates/core/ui/component/src/parser/styles.rs`
- `crates/core/ui/component/src/parser.rs`
- `crates/core/ui/elements/src/style/types.rs`
- `crates/core/ui/elements/src/style/parse.rs`
- `.planning/phases/12-theme-animation-tokens-and-css-animations/12-PATTERNS.md`
</read_first>
<steps>
1. Extend `lower_css_rules` in `crates/core/ui/component/src/parser/styles.rs` to handle `LightningCssRule::Keyframes(_)` by extracting the keyframe name and percentage blocks.
2. Accept stop selectors only when each selector is a numeric percentage between `0%` and `100%`; reject `from` and `to` with a `ParseError::InvalidStyle` message containing `from/to keyframe aliases are not supported`.
3. Reject any keyframe block containing an unsupported property with a diagnostic containing `unsupported keyframe property '<property>'`.
4. Reject a keyframe rule that contains no transition-safe animatable declarations with a diagnostic containing `keyframes '<name>' has no supported animatable properties`.
5. Add parser tests replacing `unsupported_keyframes_rule_reports_at_rule_name` with successful percentage keyframe parsing and rejection tests for `from`, `to`, unsupported properties, and non-runnable blocks.
</steps>
<acceptance_criteria>
- `crates/core/ui/component/src/parser/styles.rs` no longer treats `LightningCssRule::Keyframes(_)` as a generic unsupported at-rule.
- `crates/core/ui/component/src/parser.rs` contains a test that parses `@keyframes pulse { 0% { opacity: 0; } 50% { opacity: 0.5; } 100% { opacity: 1; } }`.
- Parser tests assert `from` and `to` are rejected.
- Parser tests assert unsupported keyframe properties reject the entire rule.
- Parser tests assert keyframes with only unsupported properties are not registered.
</acceptance_criteria>
