# Plan 13-04 - Intentional Constrained-Width Behavior

## Objective

Make the compact-state behavior explicit and testable so the richer nav bar preserves controls while shrinking secondary status content first.

## Design Truths

- The bar already uses container-query patterns; execution should extend that path, not replace it.
- Secondary copy is the first thing to compress.
- Controls remain available in compact mode.

<threat_model>
T-13-04: Richer content can appear correct at default width but fail under compression, producing clipped status text, wrapped controls, or hidden primary actions without detection.
Mitigation: write explicit compact-state rules in `main.mesh`, reuse compact behaviors from existing small components, and add a dedicated constrained-width proof in the next plan.
</threat_model>

<steps>
1. Update the nav-bar container-query rules so secondary status text shortens, hides, or drops before any primary control disappears.
2. Reuse existing compact behaviors from `battery-button.mesh` and `meta-label.mesh` where they align with the approved compact-state strategy.
3. Keep the bar visually stable under compression: no accidental wrap into multiple rows, no orphaned accent pill with no status context, and no control overlap.
4. If the status cluster includes multiple passive text elements, define which one persists longest and encode that with explicit class-level compact-state rules.
</steps>

</acceptance_criteria>
