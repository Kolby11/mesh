# Phase 13 - UI Design Contract

> Visual and interaction contract for Phase 13 navigation-bar rendering proof. Generated by `gsd-ui-phase`, verified inline against the six UI-SPEC dimensions.

---

## Scope

Phase 13 turns the shipped `@mesh/navigation-bar` module into the milestone proof surface for the v1.2 rendering upgrade.

The approved proof is a richer shell bar, not a new dashboard and not a new product feature surface. It must prove:

- real shipped-surface typography, spacing, and CSS layout
- selective passive text on the primary surface where selection is appropriate
- existing keyboard and `:focus-visible` behavior on real controls
- intentional constrained-width behavior
- token-driven transitions plus one restrained custom keyframe moment

This UI contract does not approve new feature domains such as a clock, theme dropdown, battery popover, or expanded mixer. Those remain deferred.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | none - native `.mesh` elements only |
| Icon library | XDG icon names rendered through MESH `<icon>` |
| Font | `token(typography.family)` / Inter |
| Theme source | active shell theme tokens from `config/themes/mesh-default-dark.json` and `config/themes/mesh-default-light.json` |

### Non-Negotiable Design Rules

- Use theme tokens only: `token(color.*)`, `token(spacing.*)`, `token(radius.*)`, `token(typography.*)`, and `token(animation.*)`.
- Do not introduce raw hex colors in `.mesh` implementation files.
- Keep the navigation bar recognizably shell chrome, not a dashboard card row or marketing header.
- Preserve the existing control family: settings, volume, and theme remain compact button-first controls even as the surface gains richer status copy.
- Selection must stay opt-in and narrow, following Phase 10. Passive status text may be selectable; interactive control labels and chrome must not become generally selectable.
- The single custom keyframe proof must read as a deliberate state accent, not ambient motion spread across the whole bar.

## Surface Contracts

### Primary Navigation Bar Layout

The navigation bar is the only approved Phase 13 proof surface.

| Element | Contract |
|---------|----------|
| Overall shape | One compact status cluster plus one control cluster inside the main bar. |
| Tone | Shell-like, dense, and operational; not editorial, dashboard-like, or app-homepage styled. |
| Status cluster | Holds the passive visible text that proves typography, selection, and responsive compression. |
| Control cluster | Holds the existing interactive controls and keeps them consistently available under constrained widths. |
| Settings placement | The settings control may move to the far right so the control cluster reads as an intentional shell utility group. |
| Audio popover relationship | The top-level bar remains the proof surface; the audio popover supports the surface but does not replace it as the phase centerpiece. |

### Status Content Model

| Element | Contract |
|---------|----------|
| Primary status line | Short shell-status copy that anchors the bar visually and remains visible longer under compression. |
| Secondary status line or detail | Optional compact supporting copy that can compress, truncate, or disappear first. |
| Selection eligibility | Only passive read-only status copy may be selectable. |
| Copy tone | System/status language, not CTA language, and not pseudo-navigation link text. |
| Density | Keep content compact enough that the bar still feels like a navigation shell rather than a content panel. |

### Controls

| Element | Contract |
|---------|----------|
| Button footprint | Existing compact button scale stays the baseline. |
| Labels | If labels or values appear adjacent to controls, they remain secondary to the button-first interaction model. |
| Focus ownership | The current keyboard/focus-visible rules from Phase 11 remain canonical and must not be visually diluted. |
| Availability under width pressure | Core controls stay present after secondary status text starts collapsing. |

## Responsive Contract

The navigation bar already has a container-query path and Phase 13 must make the resulting behavior intentional.

| Width Pressure Rule | Contract |
|---------------------|----------|
| First compression step | Secondary status copy compresses, truncates, or disappears before core controls do. |
| Control preservation | Settings, volume, and theme controls remain available in the constrained-width layout. |
| Layout shift style | Compression should feel like a purposeful shell compaction, not a broken wrap or accidental overflow. |
| Text strategy | Prefer reducing secondary copy, tightening gaps, and shortening passive text before reducing control affordance. |
| Narrow-state visual balance | Even in compact mode, the bar should still read as status plus controls rather than a pile of floating icons. |

Responsive anti-goals:

- Do not hide the primary interactive controls first.
- Do not introduce a broad overflow menu just for this phase.
- Do not let selectable passive text crowd focus rings or pointer targets in compact states.

## Spacing Scale

Declared values must use existing tokens and remain multiples of 4.

| Token | Value | Usage |
|-------|-------|-------|
| `spacing.xs` | 4px | Tight icon gaps, micro-adjustments inside compact status rows |
| `spacing.sm` | 8px | Dense inline spacing between adjacent controls or short text fragments |
| `spacing.md` | 16px | Default internal padding and primary content gaps |
| `spacing.lg` | 24px | Bar inline padding and larger cluster separation |
| `spacing.xl` | 32px | Maximum comfortable cluster separation in wider states |
| `spacing.2xl` | 40px | Not required by default |
| `spacing.3xl` | 48px | Not required |
| `spacing.4xl` | 64px | Not required |

Exceptions: none

### Stable Dimensions

| Element | Contract |
|---------|----------|
| Bar height | Stay near the current compact shell height; do not expand into a tall header. |
| Button size | Preserve the existing `40px` control footprint unless a plan item proves a small justified adjustment. |
| Cluster spacing | Use `spacing.md` or `spacing.lg` between the status and control clusters in wide states; compact down to `spacing.sm` where needed. |
| Corner language | Use existing theme radii, with full-pill or rounded focus states reserved for control emphasis rather than all container edges. |

---

## Typography

Use the existing theme typography. The bar should feel crisp and operational, not decorative.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body status | `token(typography.size.sm)` or `token(typography.size.md)` / 12-14px | 400-500 | `token(typography.line-height.normal)` / 1.4 |
| Secondary label | `token(typography.size.xs)` or `token(typography.size.sm)` / 10-12px | 400-500 | 1.4 |
| Control-adjacent value | `token(typography.size.sm)` / 12px | 500-600 | 1.4 |
| Emphasis label | `token(typography.size.md)` / 14px | 600-700 | 1.4 |

Typography constraints:

- Keep letter spacing at `0`.
- Do not introduce display typography, oversized numerics, or dashboard-style metric treatment.
- The primary status line may carry stronger weight, but it must still feel like shell copy rather than a heading banner.
- Secondary text should remain readable in both default and compressed states without becoming visually louder than the controls.

---

## Color

Use semantic theme tokens, not raw hex values in implementation.

| Role | Token / Value | Usage |
|------|---------------|-------|
| Dominant (60%) | `color.surface` / default dark `#1C1B1F` | Main navigation-bar background |
| Secondary (30%) | `color.surface-container`, `color.surface-container-high`, `color.surface-container-highest` | Status chips, control surfaces, and grouped chrome |
| Accent (10%) | `color.primary`, `color.secondary-container`, `color.tertiary-container` | Focus rings, active control states, and the single keyframe proof moment |
| Text emphasis | `color.on-surface`, `color.on-surface-variant` | Primary and secondary status copy |
| Destructive | `color.error` | Not part of the default bar layout for this phase |

Accent reserved for: focused controls, current active-state emphasis, selected passive text, and one restrained keyframe-driven proof moment. Accent is not a blanket tint for the whole bar.

Color constraints:

- The bar background should remain grounded in surface tokens, not become a bright accent slab.
- Secondary passive text should use variant/on-surface tokens so the control focus states stay visually stronger.
- The keyframe proof may temporarily draw from accent-family tokens, but only on one bounded element or state.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Not applicable; this is shell chrome, not a CTA surface |
| Empty state heading | Not applicable |
| Empty state body | Not applicable |
| Error state | Short factual system wording only if a dependent service such as audio is unavailable |
| Destructive confirmation | Not applicable |
| Status copy tone | Passive operational shell language that can be read, selected, and ignored without demanding action |

Copy constraints:

- Prefer short system phrases and compact values over sentence-heavy prose.
- Avoid marketing tone, product headlines, or faux-navigation labels.
- If the audio state is surfaced in visible text, it should read as a status value rather than instruction text.
- Hover titles may stay concise and utilitarian, matching the current shipped control tone.

## Interaction Contract

| Interaction | Required Behavior |
|-------------|-------------------|
| Pointer hover on controls | Uses token-driven transitions already established in the current components. |
| Keyboard focus on controls | Preserves Phase 11 `:focus-visible` and activation semantics on the real surface. |
| Pointer selection on passive text | Allowed only on the approved passive status copy areas. |
| Pointer interaction near selectable copy | Must not make controls feel selectable or weaken pointer target clarity. |
| Volume activation | Continues to open the shipped audio popover and proves control-to-popover flow on the real surface. |
| Theme activation | Continues to toggle the shipped theme behavior; Phase 13 does not replace it with a picker. |
| Settings activation | Remains an installed control even if its placement moves within the right-side cluster. |

### Animation Contract

| Motion Area | Contract |
|-------------|----------|
| General polish | Broad subtle transitions on color, border, shape, spacing, and transform should be token-driven. |
| Keyframe proof | Exactly one clear custom keyframe moment should exist on the bar or one of its controls. |
| Motion tone | Motion should feel restrained, useful, and shell-appropriate. |
| Compact-state behavior | Responsive compression must not rely on theatrical motion to explain itself. |

Animation anti-goals:

- No ambient looping motion across the whole bar.
- No multi-element choreography that turns the bar into an animated scene.
- No keyframe proof that conflicts with focus clarity or selection readability.

## Accessibility And Diagnostics

| Area | Contract |
|------|----------|
| Readability | Primary and secondary status copy must remain readable on both default themes. |
| Focus clarity | The strongest visual emphasis for interactivity remains keyboard focus and active control state, not passive text styling. |
| Selection clarity | Selectable passive text must be visually distinct only when selected, consistent with Phase 10. |
| Role separation | Passive status copy, buttons, and the audio popover trigger should remain visually and behaviorally distinct. |
| Diagnostics | Any implementation diagnostics belong in logs/tests, not as permanent on-surface developer copy. |
