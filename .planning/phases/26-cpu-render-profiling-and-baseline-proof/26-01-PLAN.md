# Plan 26-01: Attribute retained CPU render hotspots on canonical proof surfaces

## Goal

Extend the existing profiling and benchmark path so MESH can attribute the remaining CPU render cost across the retained pipeline on the shipped proof surfaces before later optimization phases change renderer behavior.

## Tasks

- [ ] Add finer-grained retained render attribution for the CPU path, covering render-object synchronization, retained display-list rebuild/update work, display-list paint traversal, and icon/image raster work alongside the existing tree-build, restyle, layout, paint, and total-surface timings.
- [ ] Thread the new attribution through the current profiling snapshot and debug JSON so `mesh.debug.profiling` exposes the extra render stages or metrics without introducing a second benchmark system.
- [ ] Keep the five canonical benchmark scenario IDs stable: `hover`, `surface_open_close`, `pointer_update`, `keyboard_traversal`, and `backend_update`.
- [ ] Ensure the benchmark targets continue to point at the shipped proof surfaces and flows already encoded in the shell debug runtime: `@mesh/navigation-bar`, `@mesh/audio-popover`, navigation-bar audio controls, and navigation-bar focus traversal.
- [ ] Add baseline-proof support so Phase 26 execution records benchmark evidence for the five canonical scenarios on the shipped proof surfaces, including the pre-change baseline and the post-instrumentation profiling view that later phases will compare against.
- [ ] Add focused tests that prove the new profiling payload remains available only when profiling is enabled, preserves the existing scenario list, and serializes the new attribution fields for shell consumers.

## Acceptance

- Developers can inspect CPU render cost for retained render-object sync, retained display-list update, paint traversal, text shaping, and icon/image raster work through the existing profiling/debug path.
- The canonical benchmark suite still exposes exactly five stable scenarios with unchanged IDs and shipped-surface targets.
- Phase 26 records benchmark evidence for those canonical scenarios on the shipped proof surfaces in a form later phases can reuse as the optimization baseline.
- Phase 26 produces a trustworthy baseline for later optimization phases without changing rendering behavior or expanding scope into GPU or parallel rendering work.

## Verification

- `cargo fmt --check`
- `cargo test -p mesh-core-shell benchmark_snapshot_exposes_five_stable_scenarios`
- `cargo test -p mesh-core-shell profiling`
- `cargo test -p mesh-core-render`

## Risks

- Over-fragmenting profiling stages could make the payload noisier without improving actionability. Favor a small set of renderer-specific metrics that line up with later roadmap phases.
- Benchmark instrumentation must remain observational. Phase 26 should not quietly alter paint behavior, damage policy, or cache heuristics while adding attribution.
