# Plan 11-02 - Focused Key Dispatch and Default Keyboard Activation

## Objective

Turn the focused control model from Plan 11-01 into a real keyboard interaction contract: focused `keydown` / `keyup` handlers, default shell-owned activation for core controls, and regression coverage that keeps Phase 10 copy behavior intact.

## Design Truths

- Focused-element `keydown` / `keyup` is local by default; keyboard events do not become surface-wide automatically.
- Buttons activate on Enter and Space release, toggles respond to Space and may also respond to Enter, and sliders step with arrow keys.
- These behaviors are shell-owned defaults, not something every plugin author must recreate with custom handlers.
- Phase 10 copy ownership stays in place when a text selection exists.

<threat_model>
T-11-02: General keyboard dispatch could duplicate events, break input editing, or steal Phase 10 copy ownership from selected text.
Mitigation: keep one shell-owned keyboard path in `component/input.rs`, preserve the early `Ctrl+C` selection gate, add focused-handler tests for controls and inputs, and verify default activation stays tied to the focused key rather than ad hoc event bubbling.
</threat_model>

<steps>
1. Extend the keyboard event path so the focused node receives `keydown` and `keyup` handler dispatch through the existing shell request pipeline, with a structured event payload that includes the key name, current modifier booleans available on keydown, current target metadata, and surface metadata.
2. Keep input editing behavior shell-owned: `Char` still appends printable input characters, `Backspace` edits focused inputs, and focused input editing continues to clear text selection when typing intent moves away from Phase 10 selection.
3. Add default shell activation behavior on top of focused dispatch: buttons activate on Enter or Space release, switches and checkboxes toggle on Space and Enter release, and sliders step with arrow keys using the slider's current `min`, `max`, and `step` attributes or the existing runtime value when `step` is absent.
4. Add tests covering focused button activation, focused toggle activation, focused slider arrow stepping, focused input Backspace behavior, and stable event payload routing for `keydown` / `keyup`.
</steps>
<acceptance_criteria>
- Focused nodes receive `keydown` and `keyup` handler dispatch through the existing request path rather than a new side channel.
- Buttons activate on Enter and Space release, switches and checkboxes toggle on Enter and Space release, and sliders step with arrow keys.
- Focused inputs still edit via `Char` and `Backspace` without breaking the new keyboard path.
- Automated tests cover focused activation and the structured handler event payload shape.
</acceptance_criteria>
