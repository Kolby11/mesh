# Plan 11-01 - Traversal Order and Focus-Visible Foundations

## Objective

Build the shell-owned focus model Phase 11 needs: a deterministic visual-order traversal collector, real `:focus-visible` state, and Tab / Shift+Tab focus movement that survives rebuilds and keeps pointer and keyboard focus coherent.

## Design Truths

- Visual order is the default traversal model; `tabindex` is the exception path.
- Traversal skips hidden and disabled controls and wraps at the ends of the surface.
- `:focus` remains the logical focused state while `:focus-visible` becomes a modality-aware visible-focus heuristic.
- Pointer-focused text entry remains visibly focused; non-text pointer focus clears keyboard-style visible focus for that interaction.

<threat_model>
T-11-01: Focus styling and traversal order could drift apart, causing the shell to focus one control while CSS, accessibility metadata, or tests observe another.
Mitigation: derive traversal from the same final laid-out tree used by runtime annotation, add explicit `focus_visible` state, and cover same-row ordering, wrap behavior, hidden/disabled filtering, and pointer/keyboard modality transitions with regression tests.
</threat_model>

<steps>

2. Extend the shell-owned runtime annotation path in `crates/core/shell/src/shell/component/runtime_tree.rs` so rebuilt trees receive both logical focus and visible-focus state from `FrontendSurfaceComponent`, including the rule that pointer-focused text-entry controls keep visible focus while pointer-focused non-text controls do not.
3. Update `crates/core/ui/render/src/render.rs` so `switch` and `checkbox` are marked focusable in `AccessibilityInfo`, matching the keyboard target set the shell already treats as controls.
4. Add tests proving `selector_matches` no longer aliases `focus-visible` to `focused`, pointer-focused inputs remain `focus_visible`, pointer-focused non-text controls do not, and switch/checkbox accessibility metadata now exposes `focusable: true`.
</steps>

<task id="11-01-02" wave="1">
<title>Add visual-order traversal collection and wire Tab / Shift+Tab focus movement</title>
<read_first>
- `crates/core/shell/src/shell/layout.rs`
- `crates/core/shell/src/shell/component/input.rs`
- `crates/core/shell/src/shell/component.rs`
- `crates/core/shell/src/shell/component/interaction_state.rs`
- `crates/core/shell/src/shell/component/tests.rs`
- `docs/frontend/mesh-syntax.md`
</read_first>
<steps>
1. Extend `crates/core/shell/src/shell/layout.rs` with a traversal collector that scans the final laid-out tree, gathers visible enabled focus targets, sorts them by top-to-bottom row order with left-to-right ties, supports `tabindex` numeric overrides, and excludes `tabindex="-1"` from normal Tab traversal while still allowing pointer/script focus.
2. Update `crates/core/shell/src/shell/component/input.rs` so `Tab` and `Shift+Tab` move `focused_key` through that traversal list, wrap at the ends, and call both `blur` on the old key and `focus` on the new key when focus changes.
3. Keep focus state coherent with pointer behavior by clearing only visible-focus modality on non-text pointer focus while preserving logical focus, and by reusing stale-target pruning so removed nodes cannot stay focused after rebuilds.
4. Add shell regression tests for same-row ordering, hidden/disabled filtering, wrap behavior, `tabindex` override ordering, `tabindex=-1`, and blur/focus handler firing during keyboard traversal.
</steps>
<acceptance_criteria>
- `crates/core/shell/src/shell/layout.rs` exposes a traversal helper that orders targets by visual geometry and honors `tabindex` overrides.
- `Tab` and `Shift+Tab` move focus through that ordered list and wrap at the ends of the surface.
- Focus changes invoke `blur` on the previous target and `focus` on the new target.
- Tests prove hidden/disabled targets are skipped and `tabindex=-1` remains pointer/script focusable but not tabbable.
</acceptance_criteria>
