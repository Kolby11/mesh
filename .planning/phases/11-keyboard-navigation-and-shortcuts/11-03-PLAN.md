# Plan 11-03 - Surface Shortcuts and Keyboard Settings Overrides

## Objective

Add the configuration and routing layers that let plugin authors declare focused-surface shortcuts while keeping shell-global shortcuts authoritative and letting shell settings remap default keyboard actions.

## Design Truths

- Surface shortcuts are explicit and focused-surface scoped; they do not become plugin-global accelerators.
- Module settings provide defaults, shell settings provide overrides.
- Shell-global shortcuts always win.
- Remappable keyboard defaults must live in shell settings, not in hardcoded control logic.

<threat_model>
T-11-03: Shortcut configuration could bypass shell-global precedence or let inactive surfaces react to keyboard input they should never receive.
Mitigation: centralize precedence in `Shell::dispatch_wayland`, keep shortcut activation keyed to the focused surface only, and test both module-default plus shell-override merge cases explicitly.
</threat_model>

<steps>
1. Extend `ShellSettings` in `crates/core/foundation/config/src/lib.rs` with a `keyboard` section that includes concrete default bindings for button activation (`Enter`, `Space`), toggle activation (`Space`, `Enter`), slider decrement/increment (`ArrowLeft`, `ArrowDown`, `ArrowRight`, `ArrowUp`), and a per-surface shortcut override map keyed by surface id.

4. Add configuration tests proving bundled defaults load, shell settings overrides merge deterministically, and absent user overrides leave module-owned shortcut defaults intact.
</steps>

- `config/settings-default.json` contains concrete keyboard defaults rather than leaving the new section absent.
- Navigation-bar declares a concrete default shortcut for the mute action through its module-owned settings contract and switches its surface `keyboard_mode` to `on_demand`.
- Automated config tests prove defaults and overrides merge deterministically.
