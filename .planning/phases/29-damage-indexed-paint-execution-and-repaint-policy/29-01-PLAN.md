# Plan 29-01 - Damage-indexed retained paint execution and repaint-policy proof

## Objective

<objective>
Make partial-damage paints proportional to the changed region by adding retained subtree command-span metadata, selecting minimal, bounding-rect, or full-surface repaint policies from measured cost and correctness signals, and feeding the software painter an ordered filtered command input instead of always scanning the full retained command list.
</objective>

## Design Truths

- D-01: Keep damage lookup subtree-first and keyed by stable retained node identity.
- D-02: Store compact command-span metadata and aggregate bounds alongside retained subtree command ownership.
- D-04: Treat scrollbars as display-list-owned chrome that repaints with the owning subtree or viewport.
- D-05: Keep tooltip overlay work outside display-list traversal.
- D-06: Prefer owner-root or viewport-root inclusion over extra per-primitive clipping machinery.
- D-07/D-08/D-09: Select minimal, bounding-rect, or full-surface repaint as a cost-aware policy, not as a fixed smallest-rect rule.
- D-10/D-11: Preserve command ordering, clipping, and batching barriers; fall back broadly when correctness is unclear.
- D-12: Publish aggregate proof through the existing `invalidation.paint` debug payload only.

<threat_model>
T-29-01: Incorrect filtered paint execution can skip visible commands, drop scrollbars, reorder surviving commands, or underpaint regions whose clip/opacity/transform ancestry is ambiguous. A second risk is producing a "partial damage" implementation that still scans the full command list and only clips each command, which would satisfy visual correctness but not the performance goal.
Mitigation: keep span metadata tied to retained subtree ownership, preserve final display-list order when building filtered command input, include owner-subtree scrollbar commands with matching damage, keep tooltip work as a separate overlay pass, escalate ambiguous cases to bounding-rect or full-surface repaint, and expose aggregate policy/filter/fallback counters with focused render and shell tests.
</threat_model>

## Risks

- Filtered command selection can underpaint visible content if span bounds ignore transform, clip, scroll, or scrollbar geometry.
- Preserving visual order can be broken if filtered commands are rebuilt by span hash, node id, or geometry order instead of original command order.
- A partial damage implementation that still passes all commands to the painter will preserve visuals but fail the Phase 29 performance goal.
- Repaint policy thresholds can become too aggressive; broad or ambiguous cases should choose full-surface fallback first and tune later in Phase 31.
