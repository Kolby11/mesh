# Plan 15-02 - Backend Update and Command Attribution Hooks

## Objective

Instrument the shell seams that already own backend update and command traffic so provider/service timings are recorded for poll/update and command-handling stages without introducing always-on overhead.

## Design Truths

- Provider identity must be attached where backend messages already cross into the shell.
- Command-handling attribution must use the active provider for the target interface.
- Disabled-mode fast paths must remain intact.

<threat_model>
T-15-02: Backend attribution can misreport work to the wrong provider or turn profiling hooks into always-on runtime overhead if it records before provider resolution or bypasses the existing profiling-enabled gate.
Mitigation: record samples only through profiling-gated shell helpers, derive provider identity from the active runtime slot or accepted backend message metadata, and add focused tests for disabled-mode inertness and provider attribution.
</threat_model>
