<read_first>
- `modules/frontend/audio-popover/src/main.mesh`

- `modules/frontend/navigation-bar/src/components/volume-button.mesh`
- `crates/core/shell/src/shell/component/tests/integration/real_surfaces.rs`
</read_first>
<action>
1. Replace independent optimistic mute/volume branches with a small shared rule: local pending state may render immediately, but backend confirmation must clear it only when the backend reaches the requested mute/volume state.
2. Make the navigation volume button render the same pending/confirmed state model used by the popover, or derive both from a common service-backed state shape exposed through the existing frontend service data.
3. Ensure volume up/down actions update the visible slider, percent label, icon, and backend command consistently.
4. Ensure mute toggles cannot appear inverted when backend updates arrive out of order.
5. Avoid service-specific Rust command semantics; keep audio command payload behavior in Luau/provider code as established by earlier phase decisions.
</action>
<acceptance_criteria>
- Muting and unmuting update the navigation icon and popover mute label consistently.
- Volume +/- actions update the popover slider and percent label immediately and reconcile with backend updates.
- The nav control does not show the opposite mute state after a backend confirmation.
- No new Rust service-specific audio command behavior is introduced.
</acceptance_criteria>
