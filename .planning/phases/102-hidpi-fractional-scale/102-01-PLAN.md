```rust
fn scale_factor_changed(
    &mut self,
    _conn: &Connection,
    _qh: &QueueHandle<Self>,
    surface: &wl_surface::WlSurface,
    new_factor: i32,
) {
    let Some(entry) = self
        .surfaces
        .values_mut()
        .find(|entry| entry.layer_surface.wl_surface() == surface)
    else {
        return;
    };
    let new_scale = new_factor as f32;
    if (entry.scale - new_scale).abs() > f32::EPSILON {
        entry.scale = new_scale;
        entry.needs_full_redraw = true;
        tracing::info!(
            scale = new_scale,
            surface_width = entry.width,
            surface_height = entry.height,
            "scale_factor_changed: integer scale update triggered full redraw"
        );
    }
}
```

This is the **integer fallback** path per HDPI-05 — used when `wp_fractional_scale_v1` is unavailable. `wl_surface::preferred_buffer_scale` events arrive through this `CompositorHandler` callback.

fn fractional_scale_converts_120x_to_f32() {

    assert!(((120.0 / 120.0) - 1.0).abs() < f32::EPSILON);

#[test]
fn physical_dimensions_ceil_logical_times_scale() {
    // Physical = ceil(logical × scale)
    let compute_physical = |logical: u32, scale: f32| -> u32 {
        ((logical as f32 * scale).ceil() as u32).max(1)

    assert_eq!(compute_physical(1920, 1.0), 1920);
    assert_eq!(compute_physical(1920, 2.0), 3840);
    assert_eq!(compute_physical(1920, 1.5), 2880);
    assert_eq!(compute_physical(100, 1.25), 125);

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Wayland compositor → mesh-core-presentation | Protocol events including `scale_factor_changed` and `preferred_scale` arrive from the compositor over the Wayland socket. These are untrusted inputs that set scale factors used for buffer sizing and damage rect computation. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-102-01 | Tampering | `scale_factor_changed` handler in `handlers.rs` | mitigate | Malicious compositor could send extreme scale values (e.g., 0 or negative). Clamp `new_factor` to `1..=3` before casting to f32. A scale of 0 would produce a zero-size buffer (crash); a negative scale would overflow `u32`. |
| T-102-02 | Tampering | `preferred_scale` handler in `handlers.rs` | mitigate | Malicious compositor could send out-of-range `preferred_scale` values. Clamp `scale` to `60..=480` (0.5× to 4.0×) before dividing by 120.0. Values outside this range are silently ignored (no scale update). |
| T-102-03 | Denial of Service | Buffer allocation size in Plan 02 | mitigate | Scale factor of 3.0 on a 4K logical surface produces a 11520×6480 physical buffer requiring ~299 MB of SHM. Cap physical dimensions at `logical × max(scale, 4.0)` and reject buffer allocation exceeding 512 MB with `PresentationError::BufferAlloc`. |
| T-102-04 | Information Disclosure | `tracing::info!` log statements in scale handlers | accept | Scale factor values and surface dimensions are not PII or secrets. Logging them at info level is acceptable for debugging and does not disclose sensitive user data. |
</threat_model>
