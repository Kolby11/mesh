**Why no Result:**
- Opaque region is a compositing hint, not a correctness gate
- If the Wayland connection is broken, the next `present_with_damage` will surface the error
- Keeps the call site in the shell loop simple (no error propagation needed)

**Existing patterns to match:**
- `pump()` dispatches to both backends with different signatures — follow that two-branch style
- `configure()` shows how to call `bridge.method(surface_id, ...)` — match that convention

Add a `pub(crate)` method `update_opaque_region` on `LayerShellBackend`:

```rust
pub(crate) fn update_opaque_region(&mut self, surface_id: &str, opaque_rect: Option<DamageRect>)
```

Place it after `present_with_damage()` and before `surface_size()`.

**Implementation:**

1. **Look up surface:** Get the `SurfaceEntry` from `self.state.surfaces.get(surface_id)`. If no surface exists for this ID, return silently (surface may not be configured yet; opaque region will be set on next present cycle).

2. **Clear case (opaque_rect is None):** 
   - Get `wl_surface` via `entry.layer_surface.wl_surface()`
   - Call `wl_surface.set_opaque_region(None)` to clear any previously-set opaque region
   - Return

3. **Set case (opaque_rect is Some(rect)):**
   - **Validate rect:** If `rect.width == 0` or `rect.height == 0`, fall through to the clear case (empty rect is equivalent to no opaque region)
   - **Create region:** Access the `wl_compositor` through `self.state.compositor_state`. The exact method to get a `wl_compositor` proxy from `CompositorState` depends on smithay-client-toolkit's API — use whatever pattern is already established in the codebase (likely `compositor_state.wl_compositor()` or through `create_surface` patterns). Create a region with `wl_compositor.create_region(&qh)` using `self.state.qh`.
   - **Add rect:** Call `region.add(rect.x as i32, rect.y as i32, rect.width as i32, rect.height as i32)` 
   - **Set opaque region:** Call `wl_surface.set_opaque_region(Some(&region))`
   - **Destroy region:** Call `region.destroy()` — CRITICAL (PITFALLS.md Pitfall 2: wl_region objects MUST be destroyed in the same frame; Wayland protocol copy semantics mean the compositor has already copied the region data at this point)

4. **Surface lookup pattern (from existing code):**
   ```rust
   let Some(entry) = self.state.surfaces.get(surface_id) else {
       return;
   };
   // must not be behind &mut self borrow if we need to call methods on state later
   ```

**Note on CompositorState access:** `CompositorState` is from `smithay-client-toolkit`. Check if it has a `wl_compositor()` accessor method or if you need to use it differently. The existing code uses `self.state.compositor_state.create_surface(&qh)` at backend.rs:552 — `create_region` likely follows the same pattern. If `create_region` is not directly available, access the underlying `wl_compositor` proxy and call `create_region` on it.

**PITFALLS.md compliance:**
- ✓ Pitfall 2 (wl_region leak): `region.destroy()` is called unconditionally after `set_opaque_region()` — both in the valid rect path and (implicitly, region is never created) in the clear/empty path
- Verify understanding: a debug-only `wl_region` object count assertion is NOT needed in this phase — the mandatory destroy-after-set pattern prevents leaks at the source

**Why no commit after set_opaque_region:**
The Wayland protocol processes `set_opaque_region` on the next `wl_surface.commit()`. The shell's `present_with_damage()` calls `layer_surface.commit()` inside `attach_shm_buffer()`, so the opaque region will be applied alongside the buffer attachment. No separate commit needed.
