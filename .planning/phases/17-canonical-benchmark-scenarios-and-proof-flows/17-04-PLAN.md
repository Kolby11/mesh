# Plan 17-04: Backend Correlation and Final Benchmark Proof

<objective>
Close Phase 17 by proving the backend-driven scenario links generic backend stage timing to visible frontend surface cost, and by running final focused benchmark/profiling/UI proof coverage.
</objective>

<requirements>
- `BACK-03`: Backend timing can be correlated with resulting frontend surface cost during backend-driven state updates.
- `BENCH-05`: Backend-driven state update scenario reports backend stage timing plus resulting frontend render cost.
</requirements>

<decision_coverage>
- `D-07`: backend-driven updates correlate existing `mesh.audio` provider stages with frontend surface cost.
- `D-08`: deterministic helper data is acceptable in tests, but no service-specific Rust behavior is added.
- `D-14`: tests prove each scenario is discoverable and maps to expected buckets/attribution fields.
- `D-16`: backend-stage attribution and resulting frontend/surface timing correlation satisfy `BACK-03`.
</decision_coverage>

<threat_model>
| Threat | Severity | Mitigation |
|--------|----------|------------|
| Correlation is faked with hard-coded audio-specific Rust behavior | high | Use existing `ProfilingBackendSnapshot` and `ProfilingSurfaceSnapshot`; tests assert generic fields and provider identity only. |
| Final benchmark result omits frontend render cost | high | Tests must assert backend result includes both backend stage text and surface render text. |
| Phase passes with only UI proof and no backend correlation | medium | Run both shell `benchmark` / `profiling_` tests and component `debug_inspector` tests before summary. |
</threat_model>
