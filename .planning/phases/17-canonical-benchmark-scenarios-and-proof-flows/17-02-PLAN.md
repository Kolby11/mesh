# Plan 17-02: Benchmark Launch Requests and Scenario Execution Hooks

<objective>
Add explicit debug-scoped benchmark run requests and map fixed scenario ids to generic shell actions or latest-run state so benchmark controls have real repeatable launch hooks.
</objective>

<requirements>
- `BENCH-01`: Provide explicit hover benchmark launch hook.
- `BENCH-02`: Provide explicit surface open/close benchmark launch hook.
- `BENCH-03`: Provide explicit pointer-driven update benchmark launch hook.
- `BENCH-04`: Provide explicit keyboard traversal benchmark launch hook.
</requirements>

<decision_coverage>
- `D-01`: benchmark runs are explicit debug-inspector actions.
- `D-04`: benchmark controls do not create a new profiling entrypoint and do not auto-toggle profiling.
- `D-06`: surface open/close uses existing shell surface transition semantics.
- `D-08`: deterministic helper state is allowed for tests, while user-facing scenarios stay shipped-surface based.
- `D-12`: skipped and unavailable scenarios are visible and non-fatal.
</decision_coverage>

<threat_model>
| Threat | Severity | Mitigation |
|--------|----------|------------|
| Benchmark launch events become an unrestricted shell command surface | high | Accept only fixed scenario ids and debug-scoped event/IPC names; reject unknown ids. |
| Benchmarks create service-specific Rust behavior | high | Map only generic shell request/profiling behavior; do not parse audio payloads or implement provider-specific command semantics. |
| Benchmark launch silently starts profiling | medium | Tests must assert run requests do not toggle `debug.profiling_enabled`. |
</threat_model>

<title>Handle benchmark run requests without starting profiling</title>
<read_first>

</read_first>
<action>
In `crates/core/shell/src/shell/runtime/request.rs`, handle `CoreRequest::RunDebugBenchmark { scenario_id }`.

Accepted scenario ids:

- `hover`
- `surface_open_close`
- `pointer_update`
- `keyboard_traversal`
- `backend_update`

For accepted ids, update debug-owned latest-run state or record a benchmark trigger in the existing debug snapshot path so the next `mesh.debug.benchmarks.scenarios[]` payload can show `Running`, `Complete`, `Waiting for samples`, or `Unavailable` for that id. If no dedicated state holder exists yet, add a small debug runtime field in `mesh_core_debug::DebugOverlayState` for latest requested scenario id/status. Do not persist across process restarts.

For `surface_open_close`, queue or apply existing generic shell surface behavior against `@mesh/audio-popover` using `ShowSurface` / `HideSurface` / `ToggleSurface` semantics already supported by the shell. For other scenarios, record the request as the latest run trigger and rely on existing profiling summaries for result display.

Unknown ids must publish a diagnostic message containing `unknown debug benchmark scenario`.
</action>

</verification>

- No code path in `RunDebugBenchmark` calls `toggle_profiling`.
