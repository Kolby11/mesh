# Plan 28-01 - Dirty-subtree retained paint command ownership and local reuse proof

## Objective

Stop local retained-tree changes from forcing whole-surface paint-command recollection by making retained command ownership subtree-local inside `mesh-core-render`, preserving unrelated descendant command payloads on transform/scroll/reorder updates, and keeping conservative full-surface fallbacks plus aggregate proof in the existing debug path.

## Design Truths

- D-01: Use stable retained node identity as the ownership key for local retained command segments.
- D-02: Keep the implementation inside `mesh-core-render`; shell/runtime code only observes aggregate proof.
- D-04: Transform-, scroll-, and reorder-only changes should keep unrelated descendant command payloads stable.
- D-08: Broad or ambiguous dirty summaries still take the conservative full fallback path.
- D-10: Proof remains aggregate-only in the existing retained invalidation/debug payload.

<threat_model>
T-28-01: Incorrect subtree reuse could silently preserve stale commands after reorder, clip, or transform changes, while insufficient fallback logic could corrupt ordering or hide content. A second risk is inflating scope by coupling subtree retention to damage-indexed execution or a new diagnostics channel.
Mitigation: drive local reuse from explicit dirty-node and dirty-summary signals, rebuild affected ancestor spans when ordering or clip correctness requires it, preserve an explicit full fallback path for broad or ambiguous dirty cases, and expose only aggregate reuse/rebuild/fallback counters through the existing profiling/debug payload with focused render and shell tests.
</threat_model>

## Risks

- Reorder or clip-sensitive subtree reuse could preserve stale ordering if ancestor span updates are too narrow.
- A local refresh path that still rebuilds the whole command vector would satisfy the code shape without meeting the phase goal.
- Debug proof that is too sparse could hide false-positive reuse bugs, while proof that is too detailed would expand scope into diagnostics work.
