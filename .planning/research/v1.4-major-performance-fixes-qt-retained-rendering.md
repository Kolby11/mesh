# Research: v1.4 Major Performance Fixes - Qt Retained Rendering

**Date:** 2026-05-09
**Focus:** Use Qt-style retained rendering principles to guide MESH performance work after the retained widget-tree foundation.

## Sources

- Qt Quick Scene Graph: https://doc.qt.io/qt-6/qtquick-visualcanvas-scenegraph.html
- Qt Quick Scene Graph Default Renderer: https://doc.qt.io/qt-6/qtquick-visualcanvas-scenegraph-renderer.html
- Qt Quick Scene Graph Adaptations: https://doc.qt.io/qt-6.9/qtquick-visualcanvas-adaptations.html
- QGraphicsView viewport update modes: https://doc.qt.io/qt-6/qgraphicsview.html
- QBackingStore partial updates: https://doc.qt.io/qt-6/qbackingstore.html

## Key Findings

### Qt Quick separates item state from render nodes

Qt Quick keeps a QML item tree as the UI-facing structure and synchronizes changed items into a separate scene-graph node tree. The scene graph is an independent render representation with geometry, material, transform, clip, and opacity nodes. This maps well to MESH: the retained `WidgetNode` tree should remain the author/runtime-facing tree, while a new retained render-object tree should be the paint-facing tree.

### Synchronization is the narrow mutation boundary

Qt Quick's render loop synchronizes changed items into scene-graph nodes before rendering. Custom item rendering updates existing nodes through `updatePaintNode(oldNode, ...)` rather than recreating nodes every frame. MESH should use stable `_mesh_key` identities and dirty summaries to update existing render objects, preserving render-object identity across style, layout, text, and paint mutations.

### Retained scene data enables batching and occlusion wins

Qt's scene graph knows the full set of primitives before drawing. That lets the renderer batch compatible primitives, minimize state changes, retain geometry/resources, and discard obscured work. MESH should build retained display data before a GPU backend so later GPU work can retain buffers and upload only changed geometry.

### Dirty regions are a policy choice, not always minimal

Qt's widget/graphics-view stack exposes several update policies: minimal viewport region, bounding rect of changes, full viewport, smart mode, and externally controlled updates. The lesson for MESH is not "always compute the smallest region." Damage tracking should choose a cost-aware region policy: minimal damage for sparse changes, bounding regions when region bookkeeping is more expensive than repainting, and full repaint fallback for unsupported changes.

### Partial updates need a retained backing/display layer

QBackingStore supports painting only a subregion of buffered window contents. MESH's current full rebuild/full repaint path has no equivalent retained backing store or display layer yet. Damage tracking should land together with retained display data so unchanged regions can be preserved, skipped, or clipped.

### GPU abstraction comes after retained data

Qt Quick can render through a scene-graph backend and, in Qt 6, through a rendering hardware abstraction that targets APIs such as Vulkan, Metal, Direct3D, and OpenGL. The important sequencing point for MESH is that the backend consumes retained scene/display data. Adding wgpu/Vulkan/OpenGL first would still pay unnecessary rebuild and upload cost if every frame produces brand-new paint data.

## MESH Implications

- Keep `WidgetNode` as the retained author/runtime tree; add a separate retained render-object tree.
- Treat typed dirty summaries as the contract between runtime/style/layout/paint/text/accessibility stages.
- Build render-object slots for transform, clip, opacity, geometry, material/style, text, and accessibility-facing data.
- Add retained display-list or paint-command storage before GPU work.
- Track damage from old and new bounds, but allow minimal, bounding-rect, full-surface, and disabled policies.
- Expose dirty node counts, retained render-object reuse, damage area, display-list reuse, batching counts, and skipped paint/present work through the existing debug inspector.
- Implement batching only after retained display data exists, and record barriers for clip, opacity, transform, text, and translucent overlap.

## Recommended Sequence

1. Typed dirty invalidation from the retained widget tree.
2. Incremental style and layout propagation.
3. Retained render-object/scene-graph tree.
4. Retained display list plus damage tracking.
5. Text shaping and glyph cache.
6. Typed attribute/style slots and interned identifiers.
7. Selector indexing for restyle.
8. Display-list batching.
9. GPU backend through wgpu/Vulkan/OpenGL in a later milestone.
10. Parallel paint/layout after ownership bottlenecks are removed.

## Risks

- Dirty summaries can become too broad and silently fall back to full work. Mitigation: debug metrics must show fallback reasons and dirty-type routing.
- Incremental layout can produce stale geometry if constraints are not modeled explicitly. Mitigation: keep full relayout fallback and test container-size and interaction-state paths.
- Damage tracking can cost more than repainting for many tiny changes. Mitigation: support cost-aware region policies.
- Batching can break visual ordering around clipping, opacity, transform, text, and translucent overlap. Mitigation: encode batching barriers explicitly and add visual regression tests.
