# Research: v1.3 Performance Instrumentation and Responsiveness

**Date:** 2026-05-08
**Milestone:** `v1.3` Performance Instrumentation and Responsiveness

## Research Goal

Define a milestone that improves shell responsiveness through measurable instrumentation on the existing MESH stack, not through speculative rewrites or “feels faster” tuning.

## Key Findings

### 1. The profiler has to follow real shell stages

Frame-time numbers alone will not explain why MESH feels slow or responsive. The milestone should attribute time across the stages the shell actually owns:

- input handling
- script/runtime update handling
- backend provider work
- tree build
- style/restyle
- layout
- paint
- present/commit
- redraw count
- total per-surface render time

This keeps measurements tied to actionable code paths instead of generic FPS summaries.

### 2. Backend attribution needs provider and stage detail

Aggregate backend timing is too weak for this codebase because MESH intentionally routes behavior through service providers. Profiling must distinguish at least:

- poll/update work
- command handling work
- state publish/delivery work

Those timings should be sliceable by provider/service so the shell can answer whether visible latency comes from a frontend render path, a specific backend provider, or state handoff between them.

### 3. Live rolling data is the right first profiler shape

`v1.3` should ship a debug-only live inspector rather than full trace persistence. Rolling recent samples and current aggregates are enough to:

- compare surfaces and providers during live interaction
- observe regressions on shipped surfaces
- validate before/after optimization results
- keep implementation and storage overhead bounded

Capture/replay and historical tracing can remain future work once the shell has a trusted live data model.

### 4. The inspector should run through the normal UI stack

The profiling UI should be built with `.mesh` components and surfaced through the existing debug overlay/debug path. That matters for two reasons:

- it proves the normal rendering/runtime stack can host complex diagnostics UI
- it avoids a special-case native inspector that would bypass the same responsiveness paths being measured

The milestone should still keep profiling debug-only so the extra UI and instrumentation never become baseline end-user cost.

### 5. Acceptance must be benchmarked on canonical interaction paths

The milestone needs a fixed benchmark suite so optimization work cannot drift into anecdotes. The canonical scenarios should be:

- hover transition latency
- whole-surface open/close render cost
- slider drag / pointer-driven update latency
- keyboard traversal / input-to-visible-response timing
- backend-driven state update impact on frontend responsiveness

`navigation-bar` and `audio-popover` are strong proof anchors because they already exercise hover, keyboard, dynamic state, and backend interaction, but the profiler itself should remain shell-wide.

### 6. Optimization scope should stay bounded

Once instrumentation exists, the milestone should spend the remainder of its budget on the worst measured hotspots first:

- render invalidation churn
- repeated restyle/layout work
- redundant redraws
- slow input-to-visible paths
- backend work only when it materially affects visible responsiveness

This keeps `v1.3` focused on measurable wins inside the current architecture instead of turning into a renderer or backend redesign milestone.

## Milestone Implications

- Profiling mode should be disabled by default and unreachable outside existing debug entry points.
- Roadmap phases should separate core timing hooks, attribution, inspector UI, benchmark proof flows, and optimization work.
- Requirements should make benchmark coverage and measurable improvement explicit acceptance criteria.

## Explicit Non-Goals

- GPU renderer rewrite
- full external tracing stack
- architecture-wide backend redesign
- broad UI redesign unrelated to profiling proof surfaces
